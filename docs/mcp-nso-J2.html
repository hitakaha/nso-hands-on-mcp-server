🔌 MCP

Model Context Protocol — 概要

LLMアプリケーションと外部コンテキスト・ツールをつなぐオープン標準です。

MCP仕様リビジョン 2025-03-26 に準拠しています。

MCPとは？

  • Anthropic が2024年11月に提唱したオープンプロトコルです
  • LLMアプリケーションが外部のコンテキストプロバイダーに接続する方法を標準化しています
📘 AI App の「USB-C」といわれ 1 つのコネクタで多様な機能をつなぎます。

MCPサーバーとは？

  • MCP経由で機能を公開するプログラムです
  • ローカル／リモートのいずれにも対応しています
  • モデル非依存 — どのLLMが利用されているかを関知しません

トランスポート

MCPは2種類の標準トランスポートを定義しています：

  • 📟 stdio — サーバーをサブプロセスとして起動し、stdin/stdout 経由でメッセージを送受信します
  • 🌐 Streamable HTTP — リモートサーバー向けの現行標準です（仕様 rev. 2025-03-26 以降）
ℹ️ 旧来の HTTP+SSE トランスポートは現在非推奨ですが、後方互換のため引き続きサポートしています。

アーキテクチャ

仕様が定義する3つのロールは以下のとおりです：

LLM  ⇄  Host (with MCP Client)  ⇄  MCP Server(s)
  • LLM — 任意の LLM モデル、複数使用可能です
  • Host — アプリケーション本体です（Claude Desktop、IDE など）
  • Server — ツール・リソース・プロンプトを提供します
⚠️ LLM自体はMCPを直接扱いません — すべての制御は Host が担います。

MCP サーバが公開するもの

🛠️ Tools 📚 Resources 📝 Prompts

  • 🛠️ Tools → 実行可能な関数
    （例：search_db、send_email）
  • 📚 Resources → 参照可能なデータ
    （例：ファイル、DBレコード、APIレスポンス）
  • 📝 Prompts → パラメータ付きのプロンプトのテンプレート
    （例：スラッシュコマンド、ワークフロー）

MCPなしでの LLM 実装

従来のループ（ユーザー ↔ モデルのみ）です。

flowchart LR U([👤 ユーザー]) -- "プロンプト" --> L[[🧠 LLM]] L -- "応答" --> U

MCPなしの制約

  • ❌ LLMは学習データの範囲しか知りません
  • ❌ リアルタイム・非公開データにはアクセスできません
  • ❌ 現実世界への操作・アクションはできません
  • ✅ 汎用的なQ&Aや推論には引き続き有効です

MCPありでの LLM 実装

ステップ1：ディスカバリ。HostのMCP Clientがサーバーに利用可能な ツール等を問い合わせます。

sequenceDiagram autonumber participant U as 👤 ユーザー participant H as 🖥️ Host (MCP Client) participant M as 🔌 MCP Server participant L as 🧠 LLM U->>H: プロンプト送信 H->>M: tools/list M-->>H: ツール定義（名前・スキーマ） H->>L: プロンプト＋ツールスキーマ

ツール呼び出しループ

ステップ2：LLMがツール呼び出しを要求し、HostがMCP経由で実行します。

sequenceDiagram participant H as 🖥️ Host participant M as 🔌 MCP Server participant L as 🧠 LLM loop LLMがツール呼び出しを終了するまで L-->>H: tool_use: 名前＋引数 H->>M: tools/call (name, args) M-->>H: ツール実行結果 H->>L: ツール結果 end L-->>H: 最終回答

重要なポイント：2つのプロトコル

  • LLMはReAct: Reasoning（思考）- Acting（行動）スタイルのプロンプト等を使用します
  • MCPサーバーはプロトコルとしてMCPを使用します
  • Hostがその間の変換・仲介を担います
💡 ポイント：Hostさえ対応していれば、同じMCPサーバーをどのLLMとも組み合わせられます。

プリミティブ — 誰が制御するか？

プリミティブ 制御主体 トリガー
🛠️ Tools LLM モデル LLMが呼び出しを判断
📚 Resources アプリケーション Hostがコンテキストとして読み込み・注入
📝 Prompts ユーザー ユーザーがテンプレートを選択

これらの制御モデルは仕様の設計意図に基づいています。

MCPが重要な理由

  • ✅ 標準化 — サーバーを一度実装すれば多くのHostで再利用できます
  • ✅ 組み合わせ可能 — 1セッション内で複数のサーバーを同時利用できます
  • ✅ 役割の明確化 — モデルが推論、サーバーが機能提供、Hostが制御を担います
  • ✅ 監査可能 — その場限りの組み合わせではなく、明確に定義された方法です

よくあるMCPの間違い

  • ❌ LLM APIの代替ではありません
  • ❌ それ単体ではサンドボックスやセキュリティ境界にはなりません
  • ❌ 特定のモデルベンダーへの依存はありません
  • ❌ 魔法ではありません — 制御の実装は Host が担います

AI によるネットワーク制御

🚀 NSO が MCP サーバとして機能します

NSO 6.7での新機能です

必要な構成

MCPを使うための各役割は下記のようになります：

  • 🔌 MCP Server → Cisco NSO
    NSOのデータ・操作を Tools/Resources として公開します
  • 🖥️ Host + MCP Client → Streamlit UIのカスタムPythonアプリ
  • 🧠 LLM → Cisco AI（デモ用LLM）
📘 同じMCPアーキテクチャに、弊社の各コンポーネントを接続した構成です。

アーキテクチャ図

flowchart LR U([👤 ユーザー]) subgraph HOST["🖥️ Host — Streamlit Python App"] UI[Streamlit UI] C[MCP Client] end L[[🧠 Cisco AI LLM]] subgraph NSO["🔌 Cisco NSO — MCP Server"] T[🛠️ Tools] R[📚 Resources] P[📝 Prompts] end U -- プロンプト --> UI UI --> C C <-- プロンプト＋ツール --> L C <-- JSON-RPC / MCP --> NSO

Cisco AIが推論し、Streamlit Hostが制御、NSOがネットワーク機能を提供します。

リクエストフロー

sequenceDiagram autonumber participant U as 👤 ユーザー participant S as 🖥️ Streamlit Host participant N as 🔌 NSO (MCP Server) participant C as 🧠 Cisco AI U->>S: ネットワークの照会・変更を依頼 S->>N: tools/list N-->>S: NSOツール一覧（例：get_device, set_config） S->>C: プロンプト＋ツール loop ツール呼び出しループ C-->>S: tool_use (name, args) S->>N: tools/call N-->>S: NSOデータ／操作結果 S->>C: ツール結果 end C-->>S: 最終回答 S-->>U: Streamlit UIで結果を表示

この構成が強力な理由

  • ✅ NSOのコアを変更せずにAI対応を実現できます
  • ✅ Cisco AIが実際のネットワーク状態を参照・操作できます
  • ✅ Streamlit Hostはシンプルなまま — 制御とUIのみを担います
  • ✅ 標準プロトコルにより、LLMの差し替えやMCPサーバーの追加が容易です
